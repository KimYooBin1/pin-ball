# 핀볼 사이트 개발 문서

## 1. 문서 목적

이 문서는 [Marble Roulette](https://lazygyu.github.io/roulette/)를 기준으로, 동일한 UX와 기술 구조를 가진 "핀볼형 추첨 사이트"를 개발하기 위한 분석 문서다.

중요한 전제:

- 원본 사이트는 전통적인 플리퍼 핀볼 게임이 아니라, "입력한 이름이 적힌 구슬이 핀볼형 맵을 따라 떨어지며 순위를 결정하는 룰렛"에 가깝다.
- 따라서 그대로 재현하려면 `실시간 물리 시뮬레이션 + 카메라 추적 + 우승 순위 지정 + 맵 교체 + 오버레이 UI`를 함께 구현해야 한다.

조사 기준:

- 서비스 URL: [https://lazygyu.github.io/roulette/](https://lazygyu.github.io/roulette/)
- 공개 저장소: [https://github.com/lazygyu/roulette](https://github.com/lazygyu/roulette)
- 조사 시점: 2026-03-31
- 저장소 최신 확인 커밋: `df76e48c0d542d3af10dfed55d461349e9fbfd0a` (2026-03-04)

## 2. 벤치마크 서비스 한 줄 요약

원본 서비스는 "이름 목록을 입력하고, 각 이름을 구슬로 생성한 뒤, 물리 기반 핀볼 맵에서 떨어뜨려 특정 순위의 구슬을 우승자로 선정하는 Canvas 기반 웹앱"이다.

핵심 인상은 다음과 같다.

- 전체 화면은 게임 Canvas가 차지한다.
- 설정 패널은 화면 좌하단에 고정된다.
- 시작 전에는 입력과 옵션 조정이 가능하고, 시작 후에는 패널이 숨겨진다.
- 카메라가 우승 후보 구슬을 따라가며, 결승선 근처에서는 줌인과 슬로우다운이 걸린다.
- 우측에는 실시간 순위 텍스트가 쌓인다.
- 좌상단에는 미니맵이 존재하며, 포인터로 카메라를 강제로 움직일 수 있다.
- 결과가 나오면 축하 파티클과 우승자 패널이 나타난다.

## 3. 사용자 경험 흐름

### 3.1 시작 전

사용자는 텍스트 영역에 이름을 쉼표 또는 줄바꿈으로 입력한다. 입력 예시는 아래와 같다.

```text
수박*2
키위
귤/3
```

지원 문법:

- `이름`
- `이름*3` : 동일 이름 구슬 3개 생성
- `이름/3` : 가중치 지정
- `이름/3*2` : 가중치 3, 개수 2

추가 동작:

- `blur` 시 중복 이름이 자동 정리된다.
- 동일 이름이 여러 번 입력되면 `*개수` 형식으로 압축된다.
- URL의 `?names=` 파라미터로 외부에서 이름을 주입할 수 있다.
- 마지막 입력값은 `localStorage.mbr_names`로 저장된다.

### 3.2 시작

`Shuffle` 버튼은 단순 셔플 버튼이라기보다 "구슬 재생성 및 시작 준비" 역할에 가깝다.

동작 순서:

1. 입력 텍스트를 파싱한다.
2. 구슬 배열을 새로 만든다.
3. 선택된 맵을 다시 로드한다.
4. 우승 순위를 다시 계산한다.
5. 시작 가능한 상태를 `ready = true`로 만든다.

`Start`를 누르면:

1. 시작 이벤트를 전송한다.
2. 설정 패널과 홍보 버튼을 숨긴다.
3. 자동 녹화가 켜져 있으면 MediaRecorder를 시작한다.
4. 물리 월드의 구슬을 활성화한다.
5. 카메라가 구슬 추적 모드로 전환된다.

### 3.3 진행 중

진행 중에 일어나는 일:

- Box2D 물리 시뮬레이션이 고정 간격으로 돈다.
- 구슬은 중력에 따라 떨어지고, 맵의 정적/회전 장애물과 충돌한다.
- 스킬이 켜져 있으면 일부 구슬이 `Impact` 스킬을 발동해 주변 구슬을 밀어낸다.
- 화면 우측에는 도착 순서와 남은 구슬의 순위가 실시간 텍스트로 표시된다.
- 좌상단 미니맵은 전체 맵과 현재 뷰포트를 보여 준다.
- 중앙 영역을 누르고 있으면 2배속이 된다.

### 3.4 종료

구슬이 `goalY` 아래로 내려가면 완주로 판정된다.

우승 처리:

- 사용자가 "첫 번째", "마지막", 또는 "직접 입력한 n번째"를 우승자로 지정할 수 있다.
- 지정한 순위의 구슬이 완주하면 게임이 종료된다.
- 파티클이 분사된다.
- 우승자 패널이 우하단에 표시된다.
- 약 1초 후 녹화가 종료되고 mp4 파일이 내려받아진다.
- 약 3초 후 설정 패널이 다시 나타난다.

- 설정 영역이 접기/펼치기 가능한 형태로 바뀐다.
- 다크모드 행은 모바일에서 숨겨진다.

## 5. 핵심 게임 규칙과 숨은 비즈니스 로직

### 5.1 입력 파싱 규칙

이름 문자열 파서는 `name`, `weight`, `count`를 추출한다.

예시:

```text
홍길동/5*3
```

파싱 결과:

- `name = 홍길동`
- `weight = 5`
- `count = 3`

### 5.2 가중치의 실제 의미

중요:

- 원본 구현에서 `/가중치`는 Box2D의 질량이나 중력을 직접 바꾸지 않는다.
- 대신 구슬의 `스킬 발동 빈도`와 `쿨타임`에 반영된다.
- 따라서 "완전한 확률 가중치"가 아니라, 스킬에 의해 결과가 간접적으로 흔들리는 구조다.

즉, 동일 사이트를 만들고 싶다면 아래 둘 중 하나를 명확히 결정해야 한다.

1. 원본처럼 "가중치 = 스킬 편향"으로 구현한다.
2. 보다 직관적으로 "가중치 = 실제 당첨 확률"이 되도록 별도 물리/확률 로직을 설계한다.

원본과 완전히 동일하게 맞추려면 1번을 따라야 한다.

### 5.3 스킬 시스템

현재 확인된 스킬은 `Impact` 하나다.

동작:

- 쿨타임이 끝날 때 일정 확률로 발동한다.
- 발동 시 현재 구슬 주변 반경 내 다른 구슬에 충격을 준다.
- 시각적으로 원형 충격파가 잠시 그려진다.

핵심 수치:

- 쿨타임 기본 범위는 대략 `1000ms ~ 5000ms`
- 가중치가 높을수록 더 자주 발동한다
- 물리 월드에서 주변 구슬에 impulse를 적용한다

### 5.4 멈춤 방지

구슬이 일정 시간 거의 움직이지 않으면 자동으로 흔들린다.

- 기준 시간: `STUCK_DELAY = 5000ms`
- 동작: 무작위 impulse 적용

즉, 맵 설계가 조금 막혀 있어도 게임이 끝까지 진행되도록 보정한다.

### 5.5 우승 판정

원본은 단순 "가장 먼저 도착한 구슬"만 지원하지 않는다.

지원 모드:

- 첫 번째
- 마지막
- 사용자 지정 n번째

마지막 우승자 로직은 별도 예외 처리가 들어가 있다. 이는 마지막 남은 구슬을 정확히 우승자로 보여 주기 위한 안전장치다.

### 6.1 런타임 구조

실행 순서는 대략 아래와 같다.

1. `RouletteRenderer.init()`가 Canvas와 리소스를 준비한다.
2. `Box2dPhysics.init()`가 물리 월드를 만든다.
3. UIObject들(`RankRenderer`, `Minimap`, `FastForwader`)을 등록한다.
4. 기본 맵을 로드한다.
5. `requestAnimationFrame` 루프를 시작한다.
6. 루프마다 물리 업데이트, 구슬 업데이트, 효과 업데이트, 카메라 업데이트, 렌더링을 수행한다.

### 6.2 업데이트 루프

원본은 프레임마다 직접 모든 물리를 돌리는 방식이 아니라, 누적 시간을 두고 `10ms` 간격으로 고정 스텝 업데이트를 수행한다.

핵심 포인트:

- `_updateInterval = 10`
- `physics.step(interval)`
- 마블 상태 업데이트
- 파티클 업데이트
- 효과 업데이트
- UIObject 업데이트
- 이후 카메라 보간
- 마지막으로 렌더링

이 구조는 FPS가 흔들려도 물리 결과를 비교적 안정적으로 유지하는 데 유리하다.

### 6.3 물리 엔진 구조

원본의 물리 레이어는 매우 단순하고 명확하다.

- 중력: `(0, 10)`
- 구슬: 반지름 `0.25`의 동적 원형 body
- 맵 엔티티: `static` 또는 `kinematic`
- 엔티티 도형: `box`, `circle`, `polyline`

특징:

- `polyline`은 EdgeShape 여러 개로 생성된다.
- 회전 장애물은 `kinematic + angularVelocity` 조합으로 만든다.
- `life`가 있는 엔티티는 충돌 시 제거된다.
- 즉, BubblePop 같은 "깨지는 버블"이 이 방식으로 구현된다.

### 6.4 렌더링 구조

렌더러는 Canvas 2D 하나를 사용한다.

렌더 순서:

1. 배경 지우기
2. 카메라 transform 적용
3. 엔티티 렌더링
4. 스킬 효과 렌더링
5. 구슬 렌더링
6. UIObject 렌더링
7. 파티클 렌더링
8. 우승자 패널 렌더링

시각 요소 특징:

- 다크 테마가 기본
- 구슬은 HSL 기반으로 색상이 분산됨
- 우승 후보는 테두리 강조
- 다크 모드에서는 장애물에 bloom 느낌이 들어감
- 키워드 이미지 또는 고정 캐릭터 이미지를 구슬 스킨으로 사용 가능

## 7. 맵 시스템 조사 결과

원본에는 4개의 스테이지가 있다.

| 맵 | goalY | zoomY | 엔티티 수 | 구조 특징 |
| --- | ---: | ---: | ---: | --- |
| Wheel of fortune | 111 | 106.75 | 46 | 긴 수직 레일, 각진 디플렉터, 회전 바, 후반부 지그재그 골인 구조 |
| BubblePop | 83 | 78 | 59 | 다수의 회전 바와 14개 파괴형 원형 버블, 비교적 짧은 스테이지 |
| Pot of greed | 111 | 110 | 32 | 항아리 모양 대칭 구조, 회전 장애물 비중이 높고 결승부가 좁음 |
| Yoru ni Kakeru | 248 | 234.5 | 288 | 매우 긴 세로형 맵, 대량의 원형 오브젝트와 회전 바, 가장 화려하고 무거운 스테이지 |

세부 관찰:

- `Wheel of fortune`
  - 시작 구간은 세로 낙하 중심이다.
  - 중반부터 45도 계열 디플렉터가 반복된다.
  - 회전 바 6개가 포함되어 있다.
  - 기본 맵으로 사용된다.
- `BubblePop`
  - 파괴형 원형 오브젝트가 많다.
  - 원형 오브젝트는 `life` 값과 높은 반발력으로 구현된다.
  - 비교적 짧아서 결과가 빠르게 나온다.
- `Pot of greed`
  - 좌우 대칭의 "병목" 구조가 강하다.
  - 정적 벽보다 회전 장애물이 결과에 더 큰 영향을 준다.
  - 우승 순위에 따라 결승부 긴장감이 크게 드러난다.
- `Yoru ni Kakeru`
  - 사실상 롱코스 모드다.
  - 정적 벽 2개로 큰 외곽을 만들고 내부에 수백 개 오브젝트를 배치한다.
  - 모바일이나 저사양 환경에서는 가장 무거운 맵이 될 가능성이 높다.

## 12. 개발 시 주의사항

### 12.1 성능

- `Yoru ni Kakeru`급 대형 맵은 엔티티 수가 많아 모바일 성능 이슈가 생길 수 있다.
- Canvas 2D만으로도 구현 가능하지만, draw call과 텍스트 렌더링 최적화가 필요하다.
- 구슬 이름 텍스트는 많은 수의 구슬이 동시에 보일 때 비용이 커질 수 있다.

### 12.2 녹화 호환성

- 원본은 `MediaRecorder`와 `captureStream()`을 사용한다.
- 브라우저별 mp4 지원이 완전히 동일하지 않을 수 있다.
- 실제 서비스에서는 WebM fallback 또는 지원 브라우저 안내가 필요하다.

### 12.3 서비스워커 경로

- 원본은 서비스워커 등록 경로가 `/roulette/service-worker.js`로 고정되어 있다.
- 신규 프로젝트의 배포 경로가 다르면 반드시 수정해야 한다.

### 12.4 외부 API 의존성

- 원본은 키워드 이미지와 일부 운영 기능을 외부 도메인에 의존한다.
- 동일한 사이트를 자체 서비스로 운영할 경우, 이 부분은 내부 API 또는 정적 자산으로 대체하는 것이 안전하다.

### 12.5 저작권/브랜딩

- 맵 이름, 캐릭터 스킨, 외부 샵 링크, 이미지 자산을 그대로 복사하는 것은 별도 검토가 필요하다.
- 동일한 동작을 목표로 하더라도, 서비스 브랜딩과 아트 자산은 자체 제작을 권장한다.

## 13. 결론

원본 사이트를 그대로 재현하려면 단순한 프론트엔드 페이지가 아니라 아래 4가지를 동시에 갖춘 웹앱을 만들어야 한다.

- 실시간 물리 기반 핀볼 엔진
- 입력 중심 추첨 UX
- 카메라/미니맵/순위 패널이 포함된 오버레이 UI
- 결과 연출과 녹화까지 포함한 운영 기능

가장 중요한 구현 포인트는 다음 두 가지다.

1. `maps.ts` 같은 선언형 맵 정의 구조를 먼저 만든다.
2. `roulette.ts`처럼 게임 루프, 물리, 카메라, 우승 판정을 하나의 오케스트레이터가 관리하게 만든다.

이 방식으로 가면 원본과 매우 비슷한 사이트를 가장 안정적으로 만들 수 있다.
